# 프로토타입 템플릿

## 이 프로젝트는

피그마 와이어프레임을 기반으로 실제로 동작하는 프로토타입을 만들어 플로우와 인터랙션을 직접 사용해보고 문제점을 찾는 용도다.
실제로 사용해볼 수 있는 수준의 디자인 퀄리티가 필요하다.

---

## 기술 스택

- **프레임워크**: React + Vite
- **언어**: JavaScript
- **스타일링**: Tailwind CSS + CSS 변수 (tokens.css)
- **폰트**: Pretendard (src/styles/fonts.css에서 CDN 로드)
- **배포**: Vercel

---

## 핵심 원칙

- 속도 우선. 빠르게 만들고, 써보고, 수정한다.
- 피그마 와이어프레임의 구조, 레이아웃, 정보 계층을 정확히 따른다. 임의로 단순화하지 않는다.
- 실제로 사용해볼 수 있는 수준의 디자인 퀄리티를 유지한다. 와이어프레임과 동일한 레이아웃 위에 tokens.css 기반으로 타이포그래피, 색상, 간격, 모서리를 다듬는다.
- 아이콘과 그래픽은 lucide-react에서 적절한 것을 사용한다. 빈칸으로 두지 않는다.
- 없는 UI는 그 자리에서 바로 만든다.
- 외부 패키지는 최소화한다. 꼭 필요한 것만 설치한다.

---

## 프로토타입 작업

- 요청(제작, 수정, 페이지 추가 등)을 받으면 먼저 요청 내용을 분석한다.
- 빠지거나 불명확하거나 오류가 있으면 질문 리스트를 한번에 출력한다.
- 모든 답변을 받은 후 작업을 시작한다. 문제없으면 바로 작업.

---

## 디자인

- 색상은 반드시 `src/styles/tokens.css`의 CSS 변수를 사용한다. (`var(--color-primary-600)`)
- tokens.css에 없는 값은 하드코딩한다.
- tokens.css에 새로운 토큰을 임의로 추가하지 않는다.
- 컴포넌트 수정 과정에서 tokens.css는 절대 건들지 않는다.
- 폰트는 Pretendard만 사용한다.
- 모바일 화면 좌우 패딩은 16px 고정.
- 모바일 기준 너비: 고정값 사용 금지. 100vw 기준으로 유동적으로 대응한다.
- 모바일 기준 높이: 고정값 사용 금지. 100dvh 기준으로 유동적으로 대응한다.
- 고정 높이 대신 min-height: 100dvh, 스크롤 가능한 레이아웃으로 구성한다.
- OS: iOS 기반
- Safe Area는 env(safe-area-inset-top), env(safe-area-inset-bottom) CSS 변수로 대응한다.
- 스크롤은 momentum scroll 적용 (-webkit-overflow-scrolling: touch)
- 탭바, 내비게이션바는 fixed로 처리하고 콘텐츠 영역은 남은 높이를 채우도록 구성한다.
- 터치 타겟은 최소 44px 이상으로 유지한다.
- 화면 전환은 iOS 네이티브 느낌으로 슬라이드 인/아웃 처리한다.
- 간격, 모서리, 그림자 등 세부 스타일도 tokens.css 값 기준으로 다듬는다.
- 타이포그래피 크기/굵기, 색상 대비, 여백은 와이어프레임을 따르되 tokens.css 기반으로 개선한다.
- 테스트 환경은 모바일 크롬(웹)이다. 아래 항목을 반드시 적용한다.
  - 텍스트 선택, 길게 누르기 등 웹 기본 동작 비활성화 (user-select: none, -webkit-tap-highlight-color: transparent)
  - 핀치 줌 비활성화 (뷰포트 메타태그에 user-scalable=no 적용)
  - 스크롤 오버스크롤 바운스 효과 비활성화 (overscroll-behavior: none)
  - 브라우저 주소창/툴바 높이 변동 대응 (100dvh 사용, 콘텐츠가 가려지지 않도록)

## 컴포넌트

- `src/components/`에 있는 공용 컴포넌트는 재활용한다.
- `src/components/`에 새로운 컴포넌트를 임의로 추가하지 않는다.
- 컴포넌트는 외부에서 완성된 것을 가져다 넣는다. 여기서 직접 만들지 않는다.
- 없는 UI는 화면 코드 안에서 직접 만든다.

---

## 피그마

- 피그마 URL을 주면 `get_design_context`로 레이아웃과 구조를 파악하고 플로우를 구현한다.
- 와이어프레임의 레이아웃, 간격, 정보 계층, 컴포넌트 배치를 최대한 동일하게 구현한다.
- 피그마에 명시된 텍스트, 수치, 구조는 그대로 반영한다. 생략하거나 합치지 않는다.
- `get_design_context`가 반환한 코드는 참고만 한다. 그대로 쓰지 않는다.

---

## 서비스 컨텍스트

- 프로젝트마다 서비스 개요 md 파일을 수동으로 추가한다.
- 파일이 있으면 반드시 읽고 서비스 도메인 용어와 구조를 파악한 후 작업한다.

---

## 파일 구조

```
src/
├── components/   ← 완성된 공용 컴포넌트 (직접 만들지 않음)
├── pages/
│   ├── v1/       ← 버전별 프로토타입
│   └── v2/
└── styles/
    ├── fonts.css  ← Pretendard CDN
    ├── index.css  ← @import fonts.css, tokens.css
    └── tokens.css ← 색상/타이포 CSS 변수
```
